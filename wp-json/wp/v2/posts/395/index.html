    Single Responsibility Principle<\/h4>\n\n\n\n
      \n
    • A class should have only one reason to change.<\/li>\n\n\n\n
    • There should be never be, more than one reason for a class to change.<\/li>\n\n\n\n
    • We should design a class in such a way that there should not be more than one responsibility associated with a class.<\/li>\n<\/ul>\n\n\n\n

      Open Closed Principle<\/h4>\n\n\n\n
        \n
      • Opening up for extension, closing it down for modifications.<\/li>\n\n\n\n
      • Modules should be opened for extension but closed for modifications.<\/li>\n<\/ul>\n\n\n\n

        Liskov Substitution Principle<\/h4>\n\n\n\n
          \n
        • Objects in a program should be replaceable with instances of their subtypes without altering the correctness of that program.<\/li>\n\n\n\n
        • Functions that use references of the base class must be able to use objects of derived class even without knowing them.\u00a0<\/li>\n\n\n\n
        • Always program in interfaces.<\/li>\n\n\n\n
        • A subclass should behave in such a way that it will not cause problems when used instead of the superclass.<\/li>\n<\/ul>\n\n\n\n

          Interface Segregation Principle<\/h4>\n\n\n\n
            \n
          • Many client-specific interfaces are better than one general-purpose interface.<\/li>\n\n\n\n
          • Clients should not be forced to depend upon interfaces that they don’t use.<\/li>\n<\/ul>\n\n\n\n

            Dependency Inversion Principle<\/h4>\n\n\n\n
              \n
            • High level modules should not depend on the lower level modules.<\/li>\n\n\n\n
            • One should depend upon Abstractions. Do not depend upon concretions.<\/li>\n\n\n\n
            • Never derive from concrete class, derive from interfaces.<\/li>\n<\/ul>\n","protected":false},"excerpt":{"rendered":"
